### 輸入上下文

#### T0032 Section 2 已證實的事實（引用不重做）

BAT 的 xterm.js wrapper **唯一入口** `src/components/TerminalPanel.tsx`，關鍵特徵：
- **沒有** `terminal.onScroll` / `onRender` / `onResize` / `onKey` / `onBell` / `onLineFeed` 訂閱
- **沒有** `buffer.active.type` 任何分支判斷（`buffer.active` 只用在 link provider 抓 line）
- `onData` 和 `attachCustomKeyEventHandler` 是 keyboard 路徑
- Context menu / wheel 是 DOM 層事件，不是走 xterm 事件
- IPC bridge：`window.electronAPI.pty.onOutput / onExit`

#### 本工單的核心問題

**「VS Code 在同樣的 xterm.js + Electron + node-pty 技術棧下，為什麼能正確處理 alt buffer 的 overlay 定位和鍵盤事件？」**

具體要回答：
1. VS Code 怎麼註冊 xterm.js event subscription？有沒有 `onScroll` / `onRender`？
2. VS Code 的 decoration / hover / inline overlay 怎麼在 scroll + alt buffer 下保持同步？
3. VS Code 的 PtyHost 架構和 BAT 的現有 IPC 設計差在哪？
4. 有沒有明確看到 `buffer.active.type === 'alt'` 的分支處理？
